# AI Agent Token Efficiency — Phased Plan

Goal: reduce the tokens AI IDE agents (Cursor, Claude Code, etc.) spend doing typical tasks on this repo. Each phase is independently valuable; the user can stop after any phase and keep the wins to date.

Observable measure for every phase: in a **fresh** agent session, perform a representative task (e.g. "fix this Vue component", "read the backend log", "look up the createNote API") and observe (a) the size of the system / rules prompt, (b) the size and number of files the agent reads to complete the task, (c) whether any single accidental `Read` or `Grep` can blow context.

Status legend: `planned` / `in-progress` / `done`.

---

## Background — `.cursorignore` vs `.cursorindexingignore`

Cursor honours two ignore files with different semantics:

- **`.cursorignore`** — total block. AI cannot index, retrieve, or read. Reserve for files the agent has **zero** legitimate reason to open: secrets, regenerable noise (`*.log`, `core`, `mysql/`, `redis/`), binary blobs.
- **`.cursorindexingignore`** — indexing-only block. Excluded from default semantic search and codebase retrieval, but the agent **can `Read` them on demand**. Reserve for content that should not pollute default retrieval but humans and AI legitimately need to consult: planning notes, long reference docs, generated API files.

Today's `.cursorignore` mixes the two. `ongoing/` and `docs/` are in `.cursorignore` even though their stated purpose is to be read on demand by humans and AI. The phased-planning skill cannot write into `ongoing/` for the same reason. This plan fixes that in Phase 1.

## Phase 1 — Block the runaway-context land mines and fix the ignore-file partitioning (done)

**Type:** Behavior. **Why first:** highest expected savings, lowest effort, lowest risk. A single accidental `Read sut.log` today wipes context; this phase makes that impossible. It also unblocks the phased-planning workflow, which currently cannot write into `ongoing/`.

**Concrete artifacts:**

Delete from working tree (all except `core` were untracked at implementation time):
- `core` (36 MB JVM crash dump)
- `sut.log` (267 MB, regenerated by `pnpm sut`)
- `e2e-full-run.log`
- `openapi-ts-error-*.log` (4 stale files from spring 2025)
- `mb.log`, `mb1.log`, `mb.pid`

Rewrite `.cursorignore` to contain only **hard-block** entries:
- `*.log`
- `core`
- `*.pid`
- `backend/logs/`
- `mysql/`, `redis/`
- `.venv*/`
- `dist/`, `**/dist/`
- existing secret patterns (keep whatever is currently there for security-sensitive files)

Create `.cursorindexingignore` (or move existing entries into it) for **retrieval-skip but readable**:
- `ongoing/`
- `docs/`
- `open_api_docs.yaml`
- `packages/generated/doughnut-backend-api/*.gen.ts` plus generated index/client entrypoints

**Verification:**
- Fresh agent session: `Grep` for a common token over the repo and confirm no log/generated files appear in results.
- `Read sut.log` is either absent (file deleted) or, if a developer regenerates it, no longer offered for retrieval.
- `Read ongoing/<existing-plan>.md` succeeds (today it returns "Permission denied").
- `Write ongoing/<new-plan>.md` succeeds via the IDE tool (today it must be done via shell).
- `Read packages/generated/doughnut-backend-api/sdk.gen.ts` succeeds when explicitly requested but does not appear in default semantic search results.

**Risks:** None material. All deleted files are auto-regenerable; the ignore split matches the documented intent of each path.

## Phase 2 — Cap log growth and give agents a safe log-tail path (planned)

**Type:** Behavior. **Why next:** Phase 1 protects retrieval, but agents legitimately need to inspect logs when debugging. Today they have only "tail it manually" warnings buried in rule files. This phase replaces good intentions with bounded files and one obvious command.

**Concrete artifacts:**
- Configure log rotation for the two logs agents actually need:
  - `sut.log` — cap ~5 MB, keep 3 backups (rotate inside `pnpm sut` setup or via a small script in `scripts/`).
  - `backend/logs/doughnut-e2e.log` — same policy via the existing Logback config.
- Add `pnpm logs:tail` (and underlying script) that prints the last 200 lines of the relevant log, with a flag to choose `sut` / `backend-e2e` / `mountebank`.
- Update `.cursor/agent-map.md` "Common Commands" to mention `pnpm logs:tail` as the default for log inspection. Remove the ad-hoc "please make sure only get tail from it" sentence in `e2e_test.mdc`.

**Verification:**
- Run the e2e suite long enough to grow the log; confirm rotation kicks in and the live file stays under cap.
- `pnpm logs:tail backend-e2e` returns ~200 lines.
- A grep for "tail" in `.cursor/rules/` returns at most one canonical reference.

**Risks:** Log rotation must not break tooling that watches the file by inode (anything tailing `sut.log`). Use copy-truncate or document any consumer changes.

## Phase 3 — Single source of truth for the always-loaded prompt (planned)

**Type:** Behavior. **Why next:** Every agent turn currently pays for the union of `CLAUDE.md` (80 lines), `general.mdc` (27 lines, `alwaysApply: true`), `planning.mdc` (91 lines, `alwaysApply: true`), and `agent-map.md` (53 lines). The Nix command prefix, `pnpm sut` assumptions, and lint commands are repeated across all four. This phase removes duplication on the per-turn hot path.

**Concrete artifacts:**
- Decide on **one** canonical home for "environment + common commands" — recommend `agent-map.md`. Make `CLAUDE.md` and `general.mdc` short pointers (≤ 15 lines each) plus only what is unique to each.
- Trim `.cursor/rules/planning.mdc` (currently 91 lines `alwaysApply: true`) to a ≤ 25-line essential checklist:
  - Phase types (Behavior vs Structure)
  - Stop-safe ordering rule
  - "No `@wip` left failing at phase end" rule
  - Pointer to the `phased-planning` skill for the long-form prose
- Move the displaced "Observable behavior first" / "Test-driven workflow" / detailed examples into the existing `phased-planning` skill (skills are loaded on demand only).
- Audit the four "lint command" duplications in `frontend.mdc`, `backend-development.mdc`, `e2e_test.mdc`, `linting_formating.mdc`; keep the canonical block in `linting_formating.mdc` only and replace the others with one-line pointers.

**Verification:**
- Compare the line count of the always-applied rule set before and after; target >= 50% reduction.
- Open a brand-new chat and confirm no two attached files state the same `nix develop -c <cmd>` recipe.

**Risks:** Losing material that one rule had and another did not. Mitigation: do the trim by extraction (move, don't delete), and review every line that disappears against `git diff`.

## Phase 4 — Split `frontend.mdc` (546 lines) into capability-scoped rule files (planned)

**Type:** Behavior. **Why next:** Today, editing any `*.vue` (even a CSS tweak) attaches all 546 lines, including Storybook setup and API integration prose. This phase makes the per-edit cost match the per-edit scope.

**Concrete artifacts (rule files named by capability, not by phase):**
- `frontend-component.mdc` — naming, structure, modals, derived state, CSS rules. Globs: `frontend/src/**/*.{vue,scss}`.
- `frontend-testing.mdc` — Vitest browser-mode, `mockSdkService`, `getByText`-over-`getByRole`, builder pattern. Globs: `frontend/tests/**/*.ts`, `frontend/src/**/*.spec.ts`.
- `frontend-api.mdc` — wrapped response pattern, `apiCallWithLoading`, `toOpenApiError`. Globs: files importing `@generated/doughnut-backend-api/sdk.gen` (or the broader `frontend/src/**/*.{ts,vue}` plus a content-based hint).
- `frontend-storybook.mdc` — story format, builders. Globs: `frontend/src/**/*.stories.ts`.
- Delete `frontend.mdc` once all sections have a new home.

**Verification:**
- Open a story file → only `frontend-storybook.mdc` and `frontend-component.mdc` attach.
- Open a `*.spec.ts` → only `frontend-testing.mdc` attaches.
- Open a CSS-only diff in a `.vue` → only `frontend-component.mdc` attaches.

**Risks:** Glob misses (e.g. component file that calls SDK). Mitigation: keep cross-references between rules small and explicit ("for API patterns see `frontend-api.mdc`").

**Verification:** same pattern as Phase 4 — confirm rule attachment scopes shrink for representative files.

**Risks:** Same as Phase 4.

## Phase 6 — Slim API summary so endpoint lookup does not require reading `sdk.gen.ts` (planned)

**Type:** Behavior. **Why now:** With the always-loaded prompt and per-area rules trimmed, the next biggest "tax per task" is API discovery. `sdk.gen.ts` (1137 lines) and `types.gen.ts` (3527 lines) are the cheapest places today to learn an endpoint shape; both are oversized for that purpose.

**Concrete artifacts:**
- Extend `pnpm generateTypeScript` to also write `packages/generated/doughnut-backend-api/api-summary.md`: one line per endpoint, format roughly `METHOD /path -> ResponseType  (request: RequestType)` plus a per-controller heading.
- The new file is generated, so it is auto-current; keep it OUT of the indexing-skip list (so it can be retrieved cheaply) while leaving `sdk.gen.ts` / `types.gen.ts` in `.cursorindexingignore`.
- Update `agent-map.md` "Generated API" section to point at `api-summary.md` as the **default** lookup, and keep `sdk.gen.ts` / `types.gen.ts` as the "exact signature" fallback.

**Verification:**
- Pick three endpoints; confirm the summary file alone is sufficient to write a correct frontend call.
- Diff `api-summary.md` after a controller signature change matches expectations.

**Risks:** Maintaining the generator. Keep it small (a single `openapi-ts` plugin or post-processing script).

## Phase 7 — Break up the largest backend test classes (planned)

**Type:** Structure (justified by ongoing test edits and Phase 8). **Why now:** With rules and API lookup trimmed, the remaining hot spot is "the file you have to open to make the change". `NotebookBooksControllerTest.java` (2327 lines) and `NotebookControllerTest.java` (1325 lines) are the worst offenders.

**Concrete artifacts (capability-named, not phase-named):**
- Split `NotebookBooksControllerTest` along its existing `@Nested` boundaries into separate files such as `NotebookBooksUploadControllerTest`, `NotebookBooksReadingControllerTest`, `NotebookBooksProgressControllerTest`, etc. — exact names derived from the current `@Nested` group names.
- Split `NotebookControllerTest` similarly (`NotebookCreationControllerTest`, `NotebookSharingControllerTest`, ...).
- Keep one shared base class only if the duplicated `@BeforeEach` / `MakeMe` setup justifies it; otherwise inline.

**Verification:**
- All backend tests still pass: `CURSOR_DEV=true nix develop -c pnpm backend:test_only`.
- No file in `backend/src/test/java/com/odde/doughnut/controllers/` exceeds ~500 lines.

**Risks:** Hidden inter-test ordering or shared mutable fixtures. Mitigation: each split file should be runnable in isolation; verify by running the new file alone.

## Phase 8 — Decompose the largest Vue components (planned)

**Type:** Structure (justified by per-edit cost on ongoing book-reading work). Only worth doing if Phases 1–6 have not already brought typical edits into a comfortable budget.

**Concrete artifacts (capability-named):**
- `PdfBookViewer.vue` (892 lines) -> extract canvas/page rendering, navigation toolbar, and OCR-readiness probe into composables under `frontend/src/composables/book-reading/`, and split sub-components such as `PdfPageCanvas.vue`, `PdfNavigationBar.vue`.
- `BookReadingContent.vue` (654 lines) -> split layout shell from the content-stream component.
- `HomePage.vue` (497 lines) -> extract recent-notes panel and quick-actions panel.
- `SearchResults.vue` (489 lines) -> extract result-row component and pagination/sort controls.

**Verification:**
- All existing frontend tests still pass: `CURSOR_DEV=true nix develop -c pnpm frontend:test`.
- No file in `frontend/src/components/` or `frontend/src/pages/` exceeds ~400 lines.
- Spot-check: a one-line UI tweak now requires opening a sub-component (~150 lines), not the whole 900-line viewer.

**Risks:** Behavior regressions in book-reading flow. Mitigation: `cypress run --spec` for the relevant book-reading feature files after each component split.

## Stop-safe summary

| After phase | What the developer keeps |
|---|---|
| 1 | No accidental megabyte reads. Plans / docs / generated API readable on demand again. Most of the win for ~15 min of work. |
| 2 | Logs are bounded; agents have one safe command for log inspection. |
| 3 | Every agent turn costs less; rule duplication eliminated. |
| 4 | Frontend edits cheap; only relevant rules attach. |
| 5 | Backend and e2e edits cheap; only relevant rules attach. |
| 6 | API lookups cheap; no need to open generated files. |
| 7 | Test edits in the worst-offender areas read 1/4 the lines. |
| 8 | Component edits in the worst-offender areas read 1/3 the lines. |

## Out of scope (intentionally)

- Reorganizing the always-applied `CLAUDE.md` content for non-Cursor tools; treat that as a follow-up if other clients become primary.
- Pruning `docs/` images and `ongoing/*.md` content — not on the hot path once Phase 1 puts them under `.cursorindexingignore`.
- Refactoring `sdk.gen.ts` itself — generated; leave it alone, work around it via Phase 6.
