## Spacing Scale

Declared values (standard set only: 4, 8, 16, 24, 32, 48, 64 — multiples of 4, matching Tailwind's existing 4px unit):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gaps inside buttons (unchanged, e.g. `Reply` icon buttons) |
| sm | 8px | `gap-2` — CTA-to-CTA / button-stack spacing inside `LinkInsertionChoice` (existing, unchanged); also `mt-2` under each matched `NoteShow` for the new "Link to this note" CTA |
| md | 16px | `gap-4` — spacing between matched-note rows in `matched-notes-section` (existing, unchanged); the new per-row CTA sits inside this same row wrapper |
| lg | 24px | `mt-6` — top margin of `matched-notes-section` (existing, unchanged) |

Exceptions (pre-existing, unchanged this phase — not part of the phase spacing scale):
- **12px (`gap-3`)** — outer stack spacing inside reused `LinkInsertionChoice`. Inherited from the existing add-link UI; this phase must not introduce new 12px tokens. Justified because forking `LinkInsertionChoice` spacing would diverge from the toolbar add-link chrome users already know (D-03).

This phase adds no new spacing token beyond `sm` (`mt-2` / 8px). The new "Link to this note" CTA is placed **inside the existing `matched-note-{id}` wrapper div**, directly below `NoteShow`, using that `mt-2` to visually associate it with the note it targets without competing with the `gap-4` (16px) rhythm between separate matched-note rows.

## Visual Hierarchy

**Primary screen:** Accidental-match recall result (`AnsweredSpellingQuestion` when `outcome === ACCIDENTAL_MATCH`).

**Focal point (Phase 4 action):** The per-matched-note **"Link to this note"** button (`daisy-btn-secondary daisy-btn-sm`, `data-testid="link-to-matched-note-{id}"`) directly under each matched `NoteShow`. This is the only new interactive affordance this phase adds; it must read as the clear next action for linking without overpowering the Phase 3 reveal content.

**Eye-path order (top → bottom):**
1. Accidental-match result alert (existing Phase 3 — outcome context)
2. Reviewed note (`NoteShow`) — source of the link (D-02)
3. "Matched note(s)" heading (`h4.text-lg.font-semibold`)
4. Each matched-note row: `NoteShow` first, then the Phase 4 CTA as the row's action
5. On CTA click: modal / `PopButton` opens `LinkInsertionChoice` (target pre-selected) — choice buttons become the dialog focal point; primary "Insert as a wiki link" stays hidden (`bareWikiLinkAvailable=false`)

**Not icon-only:** The Phase 4 CTA is a text button ("Link to this note") with matching `aria-label` / `title`. Existing go-back controls inside reused link components keep their established icon+label pattern (unchanged).

**Post-success (D-07):** Dialog closes; user remains on the same accidental-match recall result (no auto-advance). No new success banner required — both notes stay visible so the user can link another match or re-read the pair.

**New this phase — per-matched-note "Link to this note" CTA:**

| Element | Color | Rationale |
|---------|-------|-----------|
| "Link to this note" button (`data-testid="link-to-matched-note-{id}"`) | `daisy-btn-secondary daisy-btn-sm` | `daisy-secondary` is already the app's established color for the link-building family inside the add-link stack (`AddRelationshipFinalize`'s go-back button uses secondary; `LinkInsertionChoice`'s relationship option uses secondary). Reusing it for the entry-point CTA keeps one consistent "this leads to linking" color across the whole flow. `daisy-primary` stays reserved for `LinkInsertionChoice`'s own primary choice, and `daisy-accent` stays reserved for the wiki-property option per D-05 — the new CTA must not compete with either. |

Accent reserved for: the single "Add wiki link as a new property" button inside `LinkInsertionChoice` — never any other element (unchanged from pre-Phase-4 behavior; not re-litigated by this phase).

| Destructive confirmation | Not applicable — this phase has no destructive action (link creation is additive; nothing is deleted or overwritten-without-confirmation) |
