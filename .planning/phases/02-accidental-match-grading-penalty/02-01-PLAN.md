<assumption_delta_decision>
The phase goal introduces a pluralization — "matches another note's title or alias" — where previously only the reviewed note was matched against the answer. A second note now enters the answer-matching picture.

- Noun now primary: the Answer outcome identity (AnswerOutcome enum + matchedNoteId), which Phase 1 already promoted ALONGSIDE the still-required `correct: Boolean` (correct remains @NotNull and the sole SRS-credit signal).
- Decision: `add-alongside`. Phase 1 already made this call — `outcome` + `matchedNoteId` sit alongside the required `correct`; this plan is the first production writer of `outcome=ACCIDENTAL_MATCH` + `matchedNoteId`. Rationale: `correct` must stay required because existing SRS/grading and the wrong-answer threshold (RecallPromptRepository.countWrongAnswersSinceForMemoryTracker counts `quiz_answer.correct=false` rows) depend on it; the richer outcome is an addition, not a replacement.
- What would force a later promote: if `correct` ever becomes DERIVED from `outcome` (e.g. `correct = (outcome == CORRECT)`) rather than independently stored, the contract changes from "add-alongside" to "outcome replaces correct as the signal." Until then, both are stored independently and this plan writes both (correct=false AND outcome=ACCIDENTAL_MATCH) on the accidental path.
- Suggested invariant test (added in this plan's tests): an accidental match always has correct=false AND outcome=ACCIDENTAL_MATCH AND matchedNoteId != null.
</assumption_delta_decision>
